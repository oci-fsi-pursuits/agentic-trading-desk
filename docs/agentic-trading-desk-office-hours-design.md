## Terminology
- Canonical spec name: `Oracle Open Agent Specification v26.1.0`
- SDK used to author the system: `PyAgentSpec 26.1.0`
- Shorthand used in this document: `Agent Spec`
- Runtimes in scope: `WayFlow` and `LangGraph`
- UI protocols in scope: `AG-UI` is mandatory for live events, `A2UI` is optional for richer generated components

Glossary:
- `Oracle Open Agent Specification v26.1.0`: the normative agent and flow definition format for this project. Intended repo location: `spec/`.
- `PyAgentSpec 26.1.0`: the Python SDK used to author and export the spec artifacts. Intended repo location: `authoring/`.
- `WayFlow`: the Oracle-aligned runtime target used to execute the same exported spec artifact. Intended adapter location: `runtime/wayflow/`.
- `LangGraph`: the second runtime target used to execute the same exported spec artifact. Intended adapter location: `runtime/langgraph/`.
- `AG-UI`: the canonical event stream contract consumed by the frontend. Intended envelope definitions: `contracts/agui/`.
- `A2UI`: optional declarative UI payloads for charts, tables, and richer widgets. Intended render helpers: `contracts/a2ui/`.

For v1, parity means:
- artifact parity across runtimes,
- stage-order parity across runtimes,
- equivalent AG-UI event shapes for the same logical step.

Non-goals for v1:
- identical internal trace graphs,
- identical token-level reasoning,
- bit-for-bit identical transcripts.

## Problem Statement
Build a realistic, meaningful, and valuable demo for financial-services customers that mirrors the internal structure and decision-making flow of a professional buy-side investment team.

The product is not a retail trading toy and not a standards science project. It is a believable investment-committee environment where specialized agents gather evidence, debate a name, produce quantitative support, surface risk, and hand a recommendation to a human portfolio manager for approval before a trader produces an execution plan.

Portability across Agent Spec runtimes matters, but as supporting proof. The primary goal is customer belief: "yes, this is how our team could use agentic AI."

## What Makes This Cool
The whoa moment is not "look, many agents are talking." That gets old in 20 seconds.

The real delight is institutional realism:
- the desk has recognizable roles,
- disagreement is visible instead of flattened into one answer,
- the quant work shows code and metrics instead of fake authority,
- risk constraints visibly shape the decision,
- a human PM stays in control,
- the same authored desk can run on WayFlow or LangGraph without changing the front-end experience.

If this lands, a buy-side team should see a digital mirror of its own operating rhythm, not a chatbot with Bloomberg cosplay.

## Constraints
- Agent and flow definitions must follow Oracle Open Agent Specification v26.1.0 via PyAgentSpec.
- The same authored configuration must execute on WayFlow and LangGraph.
- OCI Generative AI Service is the default model backend and OCI is the default deployment target.
- The UI must use AG-UI for live interaction and can use A2UI as progressive enhancement.
- The demo should complete a trading cycle in roughly 2-5 minutes in demo mode.
- The full agent bench should exist in the spec, but the system must support selecting which agents participate in a given scenario.
- The first version must feel like a professional buy-side workflow, not a framework playground.

## Premises
1. This is a buy-side investment team simulation first. The point is to mirror how a real team gathers evidence, debates a name, sizes risk, and makes a portfolio decision.
2. "Realistic and valuable" matters more than "maximally autonomous." Human judgment stays visible at the Portfolio Manager layer.
3. The best first demo is centered on one investment decision on one security or a very small watchlist, not a market-wide autonomous platform.
4. The product should define the full agent roster, while letting each scenario control which agents are active so the workflow stays coherent.
5. The winning UI is a live investment committee room: evidence cards, visible disagreement, quant outputs, risk flags, PM approval, and trader handoff.
6. Portability between runtimes is worth proving, but only after the core buy-side workflow already feels real. For v1, the explicit target is artifact parity plus stage-order parity.

## Approaches Considered
### Approach A: Scenario-First Demo Spine
Build one polished end-to-end decision flow around a single security or thesis. Define the full agent roster, but activate only a curated subset by default with an optional seat-selection control.

Pros:
- ships fastest,
- lowest orchestration and UI complexity,
- easiest way to make the first run feel coherent.

Cons:
- partially expresses the internal team structure,
- narrower customer story,
- weaker long-term foundation for multiple desk modes.

Reuses:
- single Agent Spec flow,
- thin runtime switch layer,
- shared AG-UI event contract,
- simple participant configuration.

### Approach B: Investment Committee Simulator
Model the desk as a professional buy-side committee room. All agents exist as named roles in the spec, and each scenario has a participation matrix that determines which agents are active, optional, or suppressed during a run.

Pros:
- best match to the target customer and stated goal,
- supports the full agent bench without forcing every role into every run,
- creates a strong narrative for realism, human oversight, and evidence-backed decisions,
- scales into multiple scenarios without changing the core product frame.

Cons:
- more product design work in the UI,
- requires discipline around agent role boundaries,
- needs a shared evidence model or the screen devolves into transcript soup.

Reuses:
- full Agent Spec roster,
- scenario templates,
- participation matrix,
- common evidence objects across both runtimes.

### Approach C: Desk Composer / Demo Studio
Expose composition controls first. The user picks runtime, scenario, security, and participating agents, then watches the desk assemble and execute.

Pros:
- impressive in workshop or booth settings,
- makes agent selection and portability highly visible,
- flexible for internal exploration.

Cons:
- easier to feel like infrastructure instead of an investment-team demo,
- too many knobs for a first customer impression,
- highest risk of polishing configurability instead of value.

Reuses:
- everything from Approach B,
- plus composition UI and saved configurations.

## Recommended Approach
Choose Approach B: Investment Committee Simulator.

It best matches the stated goal: a realistic, meaningful demo for financial-services customers that mirrors how a professional buy-side team actually operates. The right abstraction is not "graph of agents" but "committee room with roles, evidence, debate, risk, and approval."

The full bench of agents should be defined in Agent Spec from day one. But participation should be scenario-driven:
- Core seats: Research Manager, Quant Analyst, Risk Manager, Portfolio Manager, Trader.
- Common research seats: Market, News, Fundamentals, Bull, Bear.
- Optional overlays: Social, Macro, Geopolitical, Aggressive, Conservative, Neutral.

This gives the user the realism of a full desk and the control to tailor participation without turning the run into noise.

### Product shape
The home screen should feel like a live investment committee meeting, not a chat app.

Primary regions:
- Business header: scenario, ticker, and business-facing status summary.
- Scenario brief: thesis, constraints, active seats, and default PM posture.
- Phase navigator: explicit tabs for the current business phase plus access to prior completed phases.
- Evidence workspace: agent outputs collapsed into evidence cards by theme, not by message order.
- Debate strip: long vs short, plus stance analysts and dissent attached to the recommendation.
- Quant notebook panel: visible code, visible outputs, confidence, and reproducible metrics.
- Risk gate + PM approval panel: hard constraints, sizing limits, and PM decision output.
- Trader handoff: trade ticket and execution plan on the primary surface.
- Monitoring panel: post-trade watch items and review cadence.
- Demo controls and diagnostics: secondary drawers for operator actions, transcript, replay, and audit context.

Wireframe artifact:
- HTML sketch: `docs/assets/agentic-trading-desk-wireframe.html`

Component inventory:
- business header with scenario selector and ticker input,
- operator drawer for runtime, replay, debate depth, and run actions,
- diagnostics drawer for transcript, audit, and simulator state,
- phase navigator with stage status,
- evidence workspace grouped by theme rather than message time,
- debate strip for long / short / neutral disagreement,
- quant notebook block with executable-code output and metrics,
- risk gate block,
- PM decision panel,
- trader handoff block,
- monitoring block.

Interaction notes:
- seat selection happens before a run starts, or via explicit rerun after changes,
- every evidence card shows provenance: role, tool or dataset, timestamp, confidence,
- dissent is attached to the recommendation object and remains visible through PM approval,
- PM edits to size, stop, or conditions force a risk re-check before the ticket can be finalized,
- in the current demo build, PM review is represented as a scripted default outcome emitted through the same approval contract; a future live mode can replace that scripted resolution without changing the UI contract.

### System shape
- Author one master desk definition in PyAgentSpec 26.1.0.
- Represent each role as its own spec component with explicit tool declarations.
- Build one canonical flow with named stages and scenario-driven participation rules.
- Add a thin orchestration layer that loads the same spec artifact and routes execution to WayFlow or LangGraph.
- Normalize runtime emissions into one AG-UI event model so the frontend does not care which runtime is underneath.
- Treat A2UI as optional rendering sugar for charts, heatmaps, and tickets. The system must remain convincing without it.

### Canonical object model
The UI cannot be built on free-form transcript output. All runtime outputs must be normalized into a small set of typed objects.

Core objects:
- `Source`: raw market, news, social, macro, or fundamentals input with provenance.
- `Evidence`: normalized fact payload derived from one or more sources.
- `Claim`: analyst assertion supported by evidence.
- `Metric`: quantitative output from the Quant Analyst.
- `Artifact`: durable run output such as notebook result, chart, summary memo, or trade ticket.
- `Constraint`: risk, mandate, or operating bound.
- `Decision`: Research Manager, Risk Manager, or Portfolio Manager conclusion.
- `TradeTicket`: final execution recommendation emitted for the Trader.

Example shapes:

```json
{
  "evidence_id": "ev_001",
  "type": "fundamentals_revision",
  "title": "EPS estimates revised upward",
  "summary": "Consensus next-quarter EPS moved +3.2% over 14 days.",
  "source_ids": ["src_earnings_014", "src_estimates_002"],
  "producer_role": "Fundamentals Analyst",
  "confidence": 0.78,
  "timestamp": "2026-03-31T15:02:10Z",
  "tags": ["fundamentals", "earnings", "nvda"]
}
```

Linking rules:
- claims link to evidence and metrics,
- decisions link to claims, dissent, and constraints,
- tickets link to decisions and constraints,
- all objects carry run ID, stage, timestamp, and provenance fields even when omitted from examples above.

### AG-UI event contract
Runtime-specific events must be translated into one canonical AG-UI stream.

Event taxonomy:
- `run.started`
- `stage.started`
- `seat.activated`
- `source.ingested`
- `evidence.upserted`
- `claim.upserted`
- `metric.upserted`
- `artifact.created`
- `approval.requested`
- `approval.resolved`
- `ticket.updated`
- `stage.completed`
- `run.failed`
- `run.completed`

Event rules:
- ordering is guaranteed within a stage, not across parallel seats,
- every event has `event_id`, `run_id`, `stage_id`, `emitted_at`, and `producer`,
- `upsert` events are idempotent on object ID,
- replay is supported by re-emitting the latest object state plus the stage timeline,
- runtime adapters must map native errors into `run.failed` or stage-level failure payloads,
- v1 parity bar: both runtimes must emit the same canonical event types for the same logical actions even if native traces differ underneath.

Formal parity comparator:
- identical stage sequence by `stage_id`,
- identical event type set per stage, order-insensitive across parallel seats,
- identical required final object set by object type and ID,
- allowed differences: timestamps, runtime-native trace IDs, and free-text summaries so long as required artifacts and decisions remain semantically equivalent.

### Participation matrix
The full desk exists in spec. Scenarios decide which seats actually participate.

Config shape:

```yaml
desk_defaults:
  Research Manager: required
  Quant Analyst: required
  Risk Manager: required
  Portfolio Manager: required
  Trader: required
  Social Analyst: optional
  Geopolitical Analyst: optional

scenario_overrides:
  earnings_committee:
    required: [Market Analyst, News Analyst, Fundamentals Analyst, Bull Researcher, Bear Researcher]
    optional: [Macro Economist, Social Analyst]
    disabled: [Geopolitical Analyst]
    conditions:
      - if: "instrument_country_risk == high"
        enable: [Geopolitical Analyst]

operator_override:
  disable: [Social Analyst]
```

Precedence and enforcement:
- precedence is `desk defaults < scenario overrides < operator override`,
- invalid overrides fail validation before a run starts,
- disabled seats cannot emit artifacts or be referenced in routing,
- if a downstream step requests an inactive seat, the orchestrator records a warning and follows the scenario fallback path instead of silently reactivating the seat.

### Stage spec
The orchestrator and UI should execute against explicit stage contracts.

| stage_id | purpose | required seats | allowed parallelism | required objects in | expected artifacts out | timeout | on_seat_failure | UI panels updated |
|----------|---------|----------------|---------------------|---------------------|------------------------|---------|-----------------|------------------|
| `gather` | Collect raw inputs and initial evidence | Market, News, Fundamentals, plus scenario-required research seats | yes | scenario, dataset refs | sources, evidence | 60s | fallback if optional seat fails, fail stage if required seat fails | phase navigator, research summary, transcript |
| `quantify` | Execute quant analysis early so quantitative validation is available before debate closes | Quant Analyst | stage contract is sequential; implementation may prefetch during gather | evidence, datasets | metrics, quant artifact | 45s | fail stage | quant notebook, research summary |
| `debate` | Generate competing claims using gathered evidence plus quant context | Bull, Bear, optional stance analysts | yes | evidence, metrics | claims, dissent links | 40s | skip failed optional seat, fail if bull or bear absent | debate strip, evidence workspace |
| `synthesize` | Produce combined recommendation | Research Manager | no | evidence, claims, metrics | decision draft | 20s | fail stage | recommendation panel |
| `risk_review` | Apply mandate and portfolio constraints | Risk Manager | no | decision draft, constraints | constraint checks, adjusted ticket draft | 20s | fail stage | risk gate |
| `pm_review` | PM review and decision capture | Portfolio Manager | no | decision draft, risk output | approval payload | demo-mode scripted by default; live mode may block for human input later | fail stage | PM action panel |
| `trade_finalize` | Produce execution plan | Trader | no | approved decision, final constraints | trade ticket | 15s | fail stage | trader handoff |
| `monitor` | Track post-decision state and hand back the monitoring plan | Trader, optional overlays | yes | ticket, watch conditions | monitoring artifact, monitoring plan | background | continue with warnings | monitor panel, transcript |

Exit criteria:
- each stage closes only when required artifacts exist or a stage-level failure is emitted,
- downstream stages consume typed objects only, never raw transcript text.

### Role charters
Core seat boundaries for v1:

- `Research Manager`
  allowed outputs: `Decision`, `Artifact(summary memo)`
  required inputs: `Evidence`, `Claim`, `Metric`
  disallowed actions: direct trade ticket emission, direct risk override
  maps to stages: `synthesize`

- `Quant Analyst`
  allowed outputs: `Metric`, `Artifact(code, chart, notebook result)`
  required inputs: dataset refs, evidence context
  disallowed actions: final recommendation, PM approval
  maps to stages: `quantify`

- `Risk Manager`
  allowed outputs: `Constraint`, `Artifact(risk memo)`, adjusted ticket draft
  required inputs: draft decision, current portfolio constraints
  disallowed actions: changing thesis rationale, bypassing PM approval
  maps to stages: `risk_review`

- `Portfolio Manager`
  allowed outputs: `Decision(approval or rejection)`
  required inputs: decision draft, risk checks, dissent links
  editable fields: size, entry conditions, exit conditions, approval notes
  disallowed actions: editing raw evidence or quant metrics
  maps to stages: `pm_review`

- `Trader`
  allowed outputs: `TradeTicket`, `Artifact(monitoring plan)`
  required inputs: approved decision, final constraints
  disallowed actions: altering approved size or thesis
  maps to stages: `trade_finalize`, `monitor`

### Quant execution model
Quant output in v1 is executed, not decorative.

Execution rules:
- sandboxed Python execution only,
- allowed libraries: `pandas`, `numpy`, `scipy`, `statsmodels`, plotting library if needed for charts,
- no unrestricted network access from the notebook environment,
- default timeout: 20 seconds per quant task,
- memory cap and CPU cap set by the execution service,
- outputs are captured as immutable artifacts: code cell input, stdout, metric objects, and chart references.

Sandbox architecture:
- containerized runner service with fixed image and preinstalled libraries,
- read-only base filesystem,
- controlled read-only dataset mounts,
- per-run working directory for generated outputs only,
- CPU, memory, and wall-time limits enforced by the runner,
- isolation boundary via hardened container runtime,
- artifact capture includes code input, stdout/stderr, exit status, derived metrics, chart files, and deterministic seed metadata.

Example starter scenarios:

Start with one: a single-name catalyst or earnings decision. That is the cleanest wedge.

### Evidence grouping and user flows
Evidence grouping rules:
- group by topic tag first (`fundamentals`, `macro`, `positioning`, `sentiment`, `risk`),
- within a topic, sort by impact and freshness,
- merge duplicate facts from multiple seats into one evidence card with multiple provenance links,
- keep dissent as a first-class linked object, not a comment thread.

Critical flows:
1. Default run:
   gather evidence -> debate -> quant -> synthesis -> risk -> PM review -> trader ticket -> monitor
2. PM edits position size:
   PM changes size or conditions -> system emits `approval.resolved` with `requires_risk_recheck=true` -> Risk Manager reruns constraint check -> ticket updates -> PM confirms again
3. Seat selection change:
   operator disables or enables optional seats before the run -> participation matrix validates the config -> stage plan updates -> run begins with revised active seat list

PM approval contract:

```json
{
  "approval_request_id": "apr_001",
  "editable_fields": ["size_bps", "entry_conditions", "exit_conditions", "approval_notes"],
  "current_size_bps": 75,
  "max_size_bps": 125,
  "constraint_ids": ["con_002", "con_004"],
  "decision_id": "dec_rm_001",
  "ticket_draft_id": "tt_draft_001"
}
```

Validation and state machine:
- PM may edit only declared editable fields,
- edited size must remain within hard limits from active constraints,
- any edit to size or conditions forces `requires_risk_recheck=true`,
- state machine:
  `approval.requested -> approval.resolved -> risk.rechecked -> ticket.finalized -> approval.confirmed`
- rejection path:
  `approval.requested -> approval.rejected -> run.completed(without_ticket)`

### Storage, replay, and retention
Storage primitives:
- `event_log`: append-only store of canonical AG-UI envelopes,
- `object_store`: materialized latest object state per object ID,
- `audit_log`: immutable security-relevant actions such as runtime choice, seat override, approval decision, and ticket generation.

Replay algorithm:
- load stage timeline from `event_log`,
- hydrate latest objects from `object_store`,
- re-render the UI from typed object state plus stage history,
- annotate replay mode in the UI to distinguish it from live execution.

Retention defaults:
- keep `audit_log` longer than conversational artifacts,
- keep object snapshots and event logs long enough for demo replay and internal review,
- purge ephemeral sandbox working directories after artifact capture completes.

## Performance Budget
The 2-5 minute target needs an explicit budget or it will drift into wishful thinking.

| Stage | Target wall time | Budget notes |
|--------|------------------|--------------|
| Gather | 30-60s | Parallel seat execution, cached market/news snapshots where possible |
| Debate | 20-40s | One round each for bull and bear in demo mode |
| Quant | 20-45s | Sandbox execution with preloaded datasets |
| Synthesis | 10-20s | Research Manager composes from typed objects, not raw transcript |
| Risk | 10-20s | Constraint evaluation over normalized ticket draft |
| PM approval | human-paced | Default demo can use scripted approval path or live operator action |
| Trader output | 5-15s | Generate ticket and monitoring plan from approved decision |

Demo-mode tactics:
- precomputed or cached datasets with timestamps,
- truncated debate rounds,
- deterministic prompts for repeatable customer demos,
- lazy-load optional seats only when enabled by scenario or operator,
- reuse fetched data across seats instead of letting every seat pull independently.

## Data Strategy
V1 should not pretend to have live institutional-grade feeds if it does not.

Recommended approach for v1:
- bundle timestamped static datasets plus curated recent-news snapshots for demo scenarios,
- label every source with freshness and provenance,
- let the UI distinguish `demo dataset` from `live integration`,
- keep an adapter boundary so licensed feeds can replace static bundles later without changing the object model.

This is the honest path. Fake “live market intelligence” in a customer demo is how trust dies.

## Minimum Security and Audit Requirements
Even as a demo, this is being shown to financial-services buyers. Sloppy security is disqualifying.

Minimum bar:
- secrets stored in OCI Vault or equivalent secret manager,
- authenticated access to scenario execution and admin controls,
- audit log for run start, seat selection, PM approval, ticket generation, and runtime choice,
- default no-PII design for scenario data,
- clear retention defaults for prompts, artifacts, and logs,
- tenant or session isolation for customer-facing demo environments,
- immutable record of PM decision and post-edit risk re-check.

## Conformance Testing
Dual-runtime parity will drift unless CI enforces it.

Required suite:
- run the same scenario in demo mode on both runtimes,
- capture canonical AG-UI event streams,

- compare stage sequence equality,
- compare event type sets per stage, order-insensitive where parallelism is allowed,
- compare required final artifacts and decisions by object type and ID,

## Reviewer Concerns
The document went through three adversarial review rounds. Most structural gaps were fixed, but these concerns remain open:

1. Normative schemas are still described in-document rather than defined as actual schema artifacts. Before implementation starts, create versioned schemas for core objects and AG-UI envelopes under a concrete `contracts/` path and validate them in CI.
2. The orchestration layer is specified by behavior, but not yet by a formal adapter interface. Before coding, define the runtime adapter API, persistence write points, and retry/error mapping rules.
3. Naming still needs one canonical cleanup pass. Stage IDs, event IDs, and role IDs should be normalized into registries so the spec, adapters, UI, and tests all use the same identifiers.
4. The parity definition is much tighter now, but still needs a machine-checkable comparator profile per object type so CI can compare stable fields and ignore non-deterministic ones.
5. V1 scope remains ambitious. If schedule pressure appears, the first cuts should be replay depth, full-bench activation in the live path, and richer A2UI work, not typed objects, PM approval, or the core committee narrative.
6. Demo mode should be made explicit before implementation. Decide the scripted PM fallback behavior, quant warm-pool strategy, and partial-metric fallback so the 2-5 minute target is operationally credible.
