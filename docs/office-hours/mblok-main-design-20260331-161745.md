## Problem Statement
Build a conference-grade, customer-facing demo for financial-services buyers where a single-ticker run produces a PM-ready decision package with risk controls. The current demo starts easily but quickly becomes unclear: short text lacks meaning, users cannot track relevance, and runs end without a clear conclusion.

## Demand Evidence
- Confirmed distribution path: sales owner and SE owner are committed to use this demo as a sales play.
- Confirmed venue: conference demo use is scheduled.
- Commercial objective: drive OCI platform consumption (compute, network, storage, GenAI), not sell the demo itself.
- Current evidence gap: no customer-side commitment logged yet (no named account workshop/POC outcome captured).

## Status Quo
Target users currently operate with fragmented workflows:
- Spreadsheet-heavy analysis
- One-off research notes
- Low standardization across roles
- Weak integration between research, quant, risk, PM decision, and execution handoff

## Target User & Narrowest Wedge
- ICP slice: L/S equity hedge fund, ~$5B AUM, ~20-person platform team.
- Buyer personas: Head of Trading, Head of Trading Platform.
- Positioning rule: primary narrative and UX defaults are optimized for L/S equity hedge funds; broader financial-services variants are deferred to post-v1 template packs.
- Narrowest wedge outcome:
  - Upload one ticker
  - Watch specialist agents contribute in real time
  - See each agent's directional lean and confidence
  - Watch collective consensus evolve
  - End with explicit PM-ready decision + risk controls + next action

## Definitions
- `PM-ready decision package`: final output containing action, sizing, horizon, top evidence, top risks, controls, and explicit next-step CTA.
- `Directional lean`: one of `Long`, `Short`, `Neutral` per agent for current stage.
- `Confidence`: calibrated 0-100 score with source role + timestamp.
- `Risk controls`: position size cap, stop-loss/invalidation trigger, and review cadence.
- `Consensus`: weighted aggregate of active agent directional lean plus confidence.

## Consensus Model (Locked for v1)
- Lean encoding: `Long=+1`, `Neutral=0`, `Short=-1`.
- Role weights (sum=1.00):
  - Market Analyst 0.12
  - News Analyst 0.10
  - Fundamentals Analyst 0.12
  - Bull Researcher 0.10
  - Bear Researcher 0.10
  - Quant Analyst 0.14
  - Risk Manager 0.16
  - Portfolio Manager 0.16
- Raw directional score:
  - `raw_score = sum(role_weight * lean_value * (confidence/100))`
- Risk adjustment:
  - If any blocking risk constraint active, multiply `raw_score` by `0.4`.
- Display mapping:
  - `score > +0.15 => Long bias`
  - `-0.15 <= score <= +0.15 => Neutral bias`
  - `score < -0.15 => Short bias`
- Worked example:
  - Pre-risk `raw_score=+0.32` (Long bias)
  - Blocking risk asserted -> adjusted score `+0.128`
  - Display changes from `Long bias` to `Neutral bias`.

## Constraints
- Must feel realistic to buy-side operating model, not a generic AI chatbot demo.
- Must preserve standards/runtime portability story, but portability is secondary to business value.
- Must support both conference narration mode and operator-driven customer workshop mode.
- Must keep run value high even if completion exceeds 2 minutes.

## Premises
1. The UI should be a decision theater, not a transcript viewer.
2. Every run must end with a hard, visible finish state: Desk Decision, Why, Risk Controls, and Next Step CTA.
3. Default experience must be progressive disclosure:
   - 5-second executive read
   - 30-second confidence/risk read
   - Deep analyst drill-down on demand

## Functional Requirements
- `FR-1`: Run State Rail must show `Now`, `Consensus`, `Risk Gate`, `Final Decision` with timestamped transitions.
- `FR-2`: Live Committee panel must show each active agent's latest directional lean, confidence, and changed/not-changed marker.
- `FR-3`: Consensus panel must visualize directional distribution and confidence delta over time.
- `FR-4`: Risk Gate panel must show pass/block status with rule-level reason codes.
- `FR-5`: Final Decision panel must include action, size, horizon, top 3 evidence, top 3 risks, controls, and next-step CTA.
- `FR-6`: Replay mode must deterministically reconstruct all agent outputs and confidence deltas.
- `FR-7`: Conference Mode must support guided reveal beats while preserving ability to switch to operator mode at any point.

## Event Contract
Event envelope fields required for every event:
- `event_id`
- `schema_version`
- `emitted_at`
- `run_seq`
- `idempotency_key`

| Event Type | Required Fields | Consumer | State Update Rule |
|---|---|---|---|
| `run_started` | `run_id`, `scenario_id`, `ticker`, `runtime` | Run State Rail | create run context and set phase to `gather` |
| `stage_transitioned` | `run_id`, `from_stage`, `to_stage`, `reason` | Run State Rail | advance active stage and append timeline checkpoint |
| `agent_started` | `run_id`, `stage_id`, `role_id` | Live Committee | mark role as in-progress for current stage |
| `agent_output` | `run_id`, `stage_id`, `role_id`, `lean`, `confidence`, `summary` | Live Committee + Evidence | update role card with latest output and append evidence row |
| `confidence_updated` | `run_id`, `stage_id`, `role_id`, `confidence`, `consensus_confidence` | Consensus panel | recalculate weighted consensus and append trend point |
| `risk_evaluated` | `run_id`, `stage_id`, `gate_status`, `constraint_results[]` | Risk Gate | set gate status `pass|warning|blocked` and render per-constraint details |
| `risk_blocked` | `run_id`, `constraint_id`, `severity`, `reason` | Risk Gate | set gate status to blocked and surface blocking rule |
| `decision_finalized` | `run_id`, `decision`, `size_bps`, `horizon`, `controls`, `top_evidence[]`, `top_risks[]`, `rationale`, `next_step` | Final Decision | lock final package and mark terminal outcome |
| `decision_package_exported` | `run_id`, `decision_id`, `export_target`, `actor` | Telemetry | increment export conversion counter |
| `workshop_cta_clicked` | `run_id`, `decision_id`, `cta_id`, `actor` | Telemetry | increment workshop CTA conversion counter |

Canonical state-machine rules:
- `run_started` -> `gather` (always first stage)
- `stage_transitioned` valid path: `gather -> quantify -> debate -> synthesize -> risk_review -> pm_review -> trade_finalize -> monitor`
- `risk_evaluated.gate_status=blocked` forces terminal decision path with `decision_finalized.decision in {REJECT, DEFER}`
- `risk_evaluated.gate_status in {pass,warning}` allows normal terminal path with `decision_finalized.decision in {APPROVE, DEFER}`
- Every run must emit exactly one terminal `decision_finalized` event.

## Risk Policy Matrix (v1)
| constraint_id | Threshold | Severity | Block Behavior | UI Message |
|---|---|---|---|---|
| `position_limit` | `size_bps > 150` | blocking | set gate blocked | `Proposed size exceeds desk limit.` |
| `liquidity` | `est_slippage_bps > 35` | warning | allow with warning | `Liquidity cost elevated, reduce size or stage entry.` |
| `event_risk` | `earnings_window=true AND size_bps > 100` | blocking | set gate blocked | `Event-risk policy requires smaller pre-event size.` |
| `correlation` | `portfolio_corr > 0.8` | warning | allow with warning | `Trade overlaps existing risk cluster.` |

Terminal decision enum semantics:
- `APPROVE`: trade ticket is authorized for execution path.
- `REJECT`: trade is blocked and no execution ticket is authorized.
- `DEFER`: trade remains unresolved pending additional analysis/human action.

## In Scope / Out of Scope
In Scope v1:
- Single-ticker flow
- Committee timeline and role-level lean visibility
- Consensus evolution
- Risk gate and final decision package
- Operator mode only (no cinematic overlay)

Out of Scope v1:
- Multi-asset portfolio construction
- OMS integration and live order routing

- Conference cinematic overlay (deferred to v2 after v1 KPI gates)

## Approaches Considered
### Approach A: Decision Spine (Minimal Viable)
Summary: Reframe current UI around a single decision spine (`Now`, `Consensus`, `Risk Gate`, `Final Decision`) while reusing existing backend/object model.

Effort: Medium  
Risk: Low

Pros:
- Fastest path to clarity
- Minimal backend change
- Quickly fixes ambiguous end-state

Cons:
- Moderate visual lift only
- May still feel incremental in conference setting

Reuses:
- Current events, object store, run/replay APIs

### Approach B: Buy-Side War Room (Ideal Architecture)
Summary: Redesign UI as a professional command center with synchronized rails for live committee state, evidence/confidence, and PM decision packet.

Effort: Large  
Risk: Medium

Pros:
- Strongest buyer credibility
- Clear role relevance and information hierarchy
- Natural path to enterprise governance/audit story

Cons:
- Larger frontend refactor
- Requires tighter event-to-state normalization

Reuses:
- Runtime adapters and contracts; mostly frontend re-architecture

### Approach C: Conference Cinematic Mode (Creative Overlay)
Summary: Add narrative mode with guided reveal beats: setup, disagreement, risk test, decision reveal, OCI architecture CTA.

Effort: Medium  
Risk: Medium

Pros:
- Strong on-stage wow and pacing
- Improves consistency across sales/SE presenters
- Converts technical complexity into business narrative

Cons:
- Needs care to avoid scripted/artificial feel
- Must coexist with unscripted operator mode

Reuses:
- Existing run engine + stage stream with presentation overlay

## Recommended Approach
Adopt a B + C hybrid:
1. Build War Room information architecture as the core product surface.
2. Layer Conference Cinematic Mode on top as optional guided narrative.

Reason:
- War Room delivers buyer trust in real operations.
- Cinematic mode delivers conference impact and repeatable storytelling.
- This pairing improves both first impression and follow-through workshop conversion.

- `SC-B3`: >= 50% of demos produce product-attributable CTA action (`decision package exported` or `workshop CTA clicked`).

Product/UX outcomes:
- `SC-P1`: In 8/10 usability sessions, users identify current phase and key decision driver in <= 5s.
- `SC-P2`: In 8/10 usability sessions, users explain why consensus changed in <= 30s.
- `SC-P3`: 100% of runs end with explicit final decision package.
- `SC-P4`: Replay reconstructs who said what, when, and why for 100% sampled runs.

System outcomes:
- `SC-S1`: Runtime switch remains possible without UI contract break.
- `SC-S2`: LLM provenance/diagnostics visible for all completed runs.
